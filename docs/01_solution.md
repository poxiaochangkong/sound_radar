# 01 — 方案选型

## 1. 背景

闭源 FPS 游戏的音频源数据(对象位置、声源元数据)**无法直接获取**。声音雷达的唯一信息来源是:游戏最终输出到 Windows 音频子系统的混音流。

可选的捕获形态有两种:
- **立体声(2.0)** — 2 个声道
- **多声道(5.1 / 7.1)** — 6 或 8 个声道

## 2. 游戏可能的三种声音处理方式

游戏在内部可能用以下三种方式之一渲染 3D 音频:

| 方式 | 原理 | 输出声道 | 方位编码 |
|---|---|---|---|
| **关闭 HRTF**(纯立体声平移) | 仅做左右能量平移 | 2.0 | 只有左右能量差,**无法区分前后** |
| **HRTF 双耳声** | 对每路声源做 HRTF 滤波,模拟 ITD/ILD/耳廓效应 | 2.0(耳机) | 左右 + 频谱 notch(编码前后/上下) |
| **基于对象的 HRTF** | 逐对象计算 HRTF 后再混音(Atmos/Sonic) | 2.0 或 7.1 | HRTF 版的增强,方位更精确 |

详细解释见 `04_audio_basics.md`。

## 3. 两种可行的雷达方案

### 方案 A:HRTF 立体声解算

让游戏开 HRTF,从立体声里**反解**方位。

- **优点**:玩家耳机听感好(原生方位化)
- **缺点**:
  - 需要逆向 HRTF(不知道游戏用的具体 HRTF 数据集)
  - 需要分析 ITD/ILD/频谱 notch
  - 算法复杂、易错、不可解释
  - 工期 2–4 周原型,1–3 月稳定

### 方案 B:多声道能量分析 ⭐ 本项目采用

让游戏输出 **7.1/5.1 离散多声道**(关闭 HRTF),从声道能量分布**直接读**方位。

- **优点**:
  - 算法极简(能量 argmax + VBAP 插值)
  - 鲁棒、可解释、无黑盒
  - 工期 1–2 周可用原型
  - 前后区分**天生可得**(有后置声道)
- **缺点**:
  - 需要游戏支持多声道输出(几乎所有现代 FPS 都支持)
  - 玩家自身耳机听感会失去 HRTF 方位感 → 需要虚拟声卡 + 二次 HRTF 虚拟化解决

## 4. 选型结论:**方案 B(多声道)**

理由:
1. 算法简单可靠,适配多款游戏的成本低
2. 前后区分是物理事实,不依赖算法技巧
3. 玩家体验缺陷可通过虚拟声卡(VoiceMeeter)+ Windows Sonic/Dolby Access 二次虚拟化补足
4. 项目目标"适用性尽可能大"在方案 B 下更易达成

## 5. 多声道方案的限制与对策

| 限制 | 对策 |
|---|---|
| 玩家耳机听感失去方位 | VoiceMeeter 虚拟声卡 + 系统级 HRTF 虚拟化(Sonic/Atmos) |
| Windows 默认设备必须配为 7.1 | 安装时给出明确指引 |
| 多声道与 HRTF 互斥 | 游戏内设置关 HRTF,系统侧补 HRTF |
| 无法识别上下方位(无高度声道) | 暂不支持,文档明确说明 |

## 6. 目标游戏兼容性

软件**不针对**任何游戏做特殊优化。验证游戏(同一份默认 config):
- 三角洲行动(Delta Force)
- CS2
- PUBG
- 无畏契约(VALORANT)

上述游戏均支持 7.1 扬声器输出,且可在设置中关闭 HRTF / 选择扬声器模式。
