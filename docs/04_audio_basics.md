# 04 — 游戏声音处理方式深入科普

本文详细解释你提到的三种游戏声音处理方式,以及它们对声音雷达的影响。

## 1. 三维声音的核心问题

人(和雷达)要定位一个声源,本质上需要判断:
- **方位角(azimuth)**:声源在你的水平面上处于哪个角度?(0°=正前)
- **仰角(elevation)**:声源在你上方还是下方?
- **距离**:多远?(本项目暂不考虑)

闭源游戏中,我们只能拿到"最终混音",所以核心问题是:**最终混音里编码了多少方位信息?编码方式是什么?**

---

## 2. 三种处理方式详解

### 2.1 关闭 HRTF(传统立体声 / 立体声平移)

**原理**:游戏仅对每个声源做**振幅平移**(amplitude panning)。
- 声源在正前方 → 左右声道音量相等
- 声源在左侧 → 左声道大、右声道小(纯能量差)
- 声源在正后方 → **同样是左右相等**(和正前方无法区分!)
- 声源在头顶 → 几乎无线索

**信息编码**:
| 方位 | 立体声能否区分? |
|---|---|
| 左 / 右 | ✅ 完美区分(L/R 能量差) |
| 前 / 后 | ❌ **无法区分**(前后镜像,能量分布一样) |
| 上 / 下 | ❌ 几乎无法 |

**对雷达的意义**:开启这种模式时,声音雷达**只能做左右雷达**,前后无法识别。这不是算法问题,是**物理上信息没编码进去**。

---

### 2.2 HRTF 双耳声(Binaural / 启用 HRTF)

**原理**:游戏对每个声源套用 **HRTF(Head-Related Transfer Function,头部相关传递函数)** 滤波器,模拟声波绕过人头、耳廓的物理过程,然后用耳机播放。

HRTF 在左右耳之间编码了三类关键线索:

1. **ITD (Interaural Time Difference, 双耳时间差)**
   - 声源在左 → 声波先到达左耳,后到达右耳
   - 低频(约 < 1.5kHz)主要靠相位差判断

2. **ILD (Interaural Level Difference, 双耳电平差)**
   - 声源在左 → 头部遮挡高频,右耳高频能量下降
   - 高频主要靠此线索

3. **频谱形变 / 耳廓 notch**
   - 高频出现的"凹陷/notch"位置编码了**仰角和前后**
   - 例如:8kHz 附近的 notch 深度指示声源在头顶还是前方

**信息编码**:
| 方位 | HRTF 能否区分? |
|---|---|
| 左 / 右 | ✅(ITD + ILD) |
| 前 / 后 | ✅(频谱 notch + 肩部反射) |
| 上 / 下 | ⚠️ 部分(耳廓效应,因人而异) |

**对雷达的意义**:理论上 HRTF 编码了 360° 方位,但**我们不知道游戏用哪套 HRTF**,逆解 HRTF 是 ill-posed 问题,算法复杂且不稳。

---

### 2.3 基于对象的 HRTF(Object-Based / Position-Based)

**原理**:游戏引擎内部知道每个声源的 **3D 世界坐标**,把每个声源作为独立"对象"实时计算 HRTF 后再混音。
- 代表技术:Dolby Atmos for Headphones、Windows Sonic、Miles Sound System
- **输出仍然是立体声**(对耳机用户),但混音过程是逐对象的、更精确
- 混响、遮挡、传播也会被建模

**和普通 HRTF 的区别**:
- 普通 HRTF 可能对"环境混响声"和"直接声"用同一处理
- 对象式 HRTF 让每个物体独立渲染,反射/遮挡也按对象几何关系算
- **方位编码更完整、更一致**

**对雷达的意义**:
- 这其实是方案 2.2 的"加强版"
- 对声音雷达**最友好**——方位编码最规范、最可学习
- 但闭源游戏中我们**无法拿到对象数据**,只能从最终混音反推

---

## 3. 多声道(5.1 / 7.1)—— 第四种、也是本项目采用的方式

这与上面三种不同,是**扬声器配置**而非"渲染方式"。游戏内需要选择"扬声器模式:7.1"。

**原理**:游戏把每个声源按方位直接平移到对应物理扬声器。

### 7.1 ITU 布局(本项目声道→角度映射)
```
           C (0°,正前)
          /
   L(-30°)── 听者 ──R(+30°)
         \              /
   Ls(-110°)         Rs(+110°)      ← 侧环绕
         \            /
   Lb(-150°)        Rb(+150°)       ← 后环绕
```

### 信息编码
| 方位 | 多声道能区分? |
|---|---|
| 左 / 右 | ✅(L/R vs Ls/Rs vs Lb/Rb) |
| 前 / 后 | ✅✅(有 Ls/Rs、Lb/Rb,**天生区分**) |
| 上 / 下 | ❌(无高度声道) |

**对雷达的意义**:
- 方位 = 声道能量 argmax + 邻接插值(VBAP)
- **算法极简,鲁棒,可解释**
- 唯一需要配合的是:让游戏输出 7.1、关闭 HRTF

---

## 4. 四种方式对比

| 处理方式 | 输出声道 | 左右 | 前后 | 上下 | 雷达实现难度 |
|---|---|---|---|---|---|
| 关闭 HRTF(立体声) | 2.0 | ✅ | ❌ | ❌ | 低(但只能左右) |
| HRTF 双耳声 | 2.0 | ✅ | ⚠️ 需逆解 | ⚠️ 部分 | 中-高 |
| 基于对象 HRTF | 2.0 或 7.1 | ✅ | ⚠️ 或 ✅ | ⚠️ 部分 | 中-高 |
| **多声道 7.1** ⭐ | 8.0 | ✅ | ✅ | ❌ | **低** |

---

## 5. 为什么本项目选多声道?

### 物理事实
当游戏输出 7.1 时:
- 声源在正前 → 游戏混音让 **C 声道**最响
- 声源在左前 → **L** 最响
- 声源在左后 → **Ls 或 Lb** 最响
- 声源在正后 → **Lb + Rb** 同时响

方位被**直接编码**为声道能量分布,不需要任何逆解。

### 算法代价对比
- 多声道:能量 argmax + VBAP → 几十行代码
- HRTF 逆解:ITD 估计 + ILD + 频谱 notch 匹配 + 不知道用哪套 HRTF → 上千行 + 不稳

### 通用性
多声道方案对游戏的依赖**最小**:
- 只要游戏支持 7.1 输出(几乎所有现代 FPS 都支持)
- 只要游戏允许关闭 HRTF / 选扬声器模式
- 雷达的算法对所有游戏**完全一致**

## 6. 多声道方案的代价与对策

| 代价 | 对策 |
|---|---|
| 玩家耳机听感失去方位 | VoiceMeeter 虚拟声卡 + 系统级 HRTF 虚拟化(Sonic/Atmos) |
| Windows 默认设备必须配为 7.1 | 安装/启动指引中明确说明 |
| 多声道与 HRTF 互斥 | 游戏内关 HRTF,系统侧补 HRTF |
| 无法识别上下方位 | 本项目暂不支持,文档明确说明 |
