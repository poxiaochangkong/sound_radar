# sound_radar 仓库分析归纳

仓库地址：https://github.com/poxiaochangkong/sound_radar

## 1. 总体结论

`sound_radar` 的设计目标是实现一个基于多声道游戏音频的“声音雷达”。它不尝试读取游戏内存、游戏 API 或敌人坐标，而是通过 Windows 音频系统捕获游戏最终输出的多声道音频流，再根据不同声道的能量分布估计声源方向，最后在雷达 UI 上显示声音事件。

从文档看，它的目标比较完整：支持 7.1 / 5.1 多声道、WASAPI loopback 捕获、声道能量分析、瞬态事件检测、方向估计、雷达显示、标定工具和游戏无关的配置体系。

从代码看，它已经实现了一个可运行原型：能捕获音频、计算特征、检测 onset、估计方向、生成雷达点并显示 UI。但它距离文档描述的完整产品还有明显差距，主要问题集中在设备自检、错误处理、算法稳定性、多声源处理和 UI 完整度上。

## 2. 文档中软件想实现的功能

### 2.1 核心思路

文档认为，对于闭源 FPS 游戏，外部程序无法直接获取游戏对象位置或声源元数据。因此，软件只能分析游戏最终输出到 Windows 音频系统的混音流。

文档比较了两种路线：

1. HRTF 立体声反解  
   尝试从耳机立体声中还原方位，需要逆向未知 HRTF、分析 ITD / ILD / 频谱 notch，复杂且不稳定。

2. 多声道能量分析  
   让游戏输出 7.1 或 5.1 多声道，关闭游戏 HRTF，然后捕获每个声道的能量。因为游戏已经把空间方位编码到了不同扬声器声道中，所以程序只需要分析各声道能量分布即可估计方向。

仓库最终选择的是第二种方案。

### 2.2 预期音频链路

文档预期的数据流大致是：

```text
Game 7.1 / 5.1 输出
        ↓
Windows / VoiceMeeter 多声道虚拟设备
        ↓
WASAPI loopback 捕获多声道 PCM
        ↓
每声道能量与频谱特征提取
        ↓
瞬态事件检测
        ↓
声源方位估计
        ↓
PyQt 雷达 UI 显示
```

### 2.3 预期功能模块

文档中设计的软件模块包括：

- 音频捕获模块：通过 WASAPI loopback 获取 Windows 输出音频。
- 特征提取模块：计算每声道 RMS、子带能量等。
- 噪声底估计模块：自适应估计环境噪声。
- onset 检测模块：用 spectral flux 检测脚步、枪声等瞬态事件。
- 方向估计模块：用多声道 SNR 加权插值估计角度。
- 平滑跟踪模块：把瞬时方向事件转换为雷达上的短暂接触点。
- UI 模块：用 PyQt 显示雷达界面、接触点和基础控制项。
- 标定工具：为不同游戏和设备生成 profile。
- 配置系统：支持默认配置、用户配置和 profile 合并。

### 2.4 设计目标

文档强调几个重要设计目标：

- 游戏无关：不依赖某个特定游戏的内存结构或 API。
- 不做 HRTF 反解：避免从复杂的双耳信号中逆向空间方位。
- 依赖多声道：要求游戏和系统实际输出 7.1 / 5.1。
- 自适应阈值：尽量通过 noise floor 和 spectral flux 自动适应环境。
- 可标定：不同游戏可通过 profile 调整阈值和参数。
- UI 可视化：把声音事件显示为雷达 blip，而不是直接给出敌人坐标。

## 3. 代码实际实现了什么

### 3.1 启动流程

`src/main.py` 是程序入口。它会读取配置，创建 PyQt 应用，创建 `MainWindow`，然后启动音频捕获和处理流程。

实际功能：

- 加载配置文件；
- 初始化 QApplication；
- 创建雷达主窗口；
- 启动音频捕获；
- 进入 PyQt 事件循环。

### 3.2 音频捕获

`src/audio/capture.py` 中的 `AudioCapture` 使用 `sounddevice.InputStream`，通过 `sd.WasapiSettings(loopback=True)` 实现 WASAPI loopback 捕获。

实际功能：

- 根据配置选择设备；
- 根据配置声道布局设置声道数；
- 捕获音频帧；
- 将音频帧通过回调传给后续处理逻辑。

它实现了基础捕获能力，但没有很好地验证实际设备声道数是否与配置一致。

### 3.3 特征提取

`src/analysis/features.py` 实现了：

- 每声道 RMS²；
- mono mixdown 后的 FFT；
- 子带能量计算。

默认频带包括：

- 0–200 Hz；
- 200–2000 Hz；
- 2000–8000 Hz；
- 8000–24000 Hz。

这说明代码已经具备基础能量分析能力。

### 3.4 噪声底估计

`src/analysis/noise_floor.py` 中的 `NoiseFloorEstimator` 使用类似 envelope follower 的方式维护每个声道的 noise floor。

实际功能：

- 每声道维护一个噪声底估计值；
- 输入能量升高时按 attack 更新；
- 输入能量降低时按 release 更新；
- 设置最小值避免除零或数值过小。

### 3.5 onset 检测

`src/analysis/onset.py` 中的 `OnsetDetector` 使用 spectral flux 检测瞬态事件。

实际功能：

- 将多声道音频混成 mono；
- 做 FFT；
- 计算相邻帧频谱的正向差分；
- 用滚动中位数构造自适应阈值；
- 触发后进入 refractory period，避免短时间内重复触发。

它能检测“是否出现突发声音”，但不能直接判断哪个声道突发，也不支持复杂多声源分离。

### 3.6 方向估计

`src/analysis/direction.py` 中的 `estimate_direction()` 会：

- 忽略默认配置中的 C 和 LFE 声道；
- 根据每声道能量和 noise floor 计算 SNR；
- 过滤低于阈值的声道；
- 找到 SNR 最高的声道；
- 再找一个角度最接近且也超过阈值的声道；
- 用两个声道的 SNR 进行角度插值；
- 输出估计方向。

这实现了一个简单的 VBAP 风格方向估计，但不是严格的声源分离算法。

### 3.7 雷达显示与平滑

`src/tracking/smoother.py` 中的 `ContactSmoother` 把瞬时方向事件转换为会衰减的雷达接触点。

实际功能：

- 相近角度的事件会合并刷新；
- 雷达点会随时间按半衰期衰减；
- 超过一定时间或强度过低后删除。

UI 使用 PyQt 绘制雷达圆、方向标签、扫描线和接触点。

### 3.8 工具脚本

仓库还提供了一些辅助工具：

- `tools/calibrate.py`：用于静音阶段和活动阶段采样，生成 profile。
- `tools/record_session.py`：录制多声道 loopback 到 WAV，用于检查实际捕获效果。

这些工具说明作者确实考虑到了设备和游戏差异，但主程序本身的自动诊断还不够完善。

## 4. 文档与代码的主要差距

### 4.1 UI 功能未完全实现

文档中提到 UI 应支持灵敏度调整和声道布局切换，例如 `layout_changed`。实际代码中主窗口主要只有灵敏度滑杆，没有完整实现声道布局下拉和对应信号。

### 4.2 设备错误提示不够完整

文档要求在设备不存在、声道数不匹配、队列丢帧等情况下给出明确提示。实际代码更多是依赖打开音频流时是否抛异常，没有完整的设备枚举、实际声道验证和 UI 错误提示。

### 4.3 异常处理过于粗糙

代码中存在多处 `except Exception: pass`。这会导致音频回调、处理线程或 UI 更新出错时，程序可能静默失败，用户无法知道具体原因。

### 4.4 onset refractory 与文档描述不一致

文档中描述的 refractory 更像是“同频段内”的抑制。代码实际只有一个全局 `_last_onset_time`，任何事件触发后都会让所有后续事件进入同一个不应期。这可能导致连续或重叠声音被漏掉。

### 4.5 方向估计不是严格的相邻扬声器对

文档中的 VBAP 思路是选择最高 SNR 声道以及它的相邻声道。代码实际是选择最高 SNR 声道和角度最近的 above-threshold 声道。多数情况下效果接近，但在复杂多声源或多个声道同时过阈值时可能不稳定。

### 4.6 默认配置兜底不完整

文档说配置优先级是 profile > config.yaml > 硬编码默认值。但代码中的硬编码默认值并不包含完整运行所需的所有字段，例如完整的 channel_angles 和 UI radar 颜色配置。因此没有配置文件时不一定能真正独立运行。

## 5. 主要缺陷分析

### 5.1 只能粗略处理单个瞬态事件

当前算法本质上假设同一时间主要只有一个显著声音事件。真实游戏中，脚步、枪声、爆炸、队友、环境音往往会重叠。多个声源同时出现时，算法通常会偏向能量更强的声源，而不是分离多个方向。

因此它更像是“声音方向提示器”，不是稳定可靠的敌人定位系统。

### 5.2 强依赖真实 7.1 / 5.1 输出

这个方案能否有效，取决于游戏、Windows、声卡、VoiceMeeter 或虚拟音频设备是否真的输出和保留了离散多声道。

如果中间链路发生以下情况，效果会明显下降：

- 游戏实际只输出 stereo；
- Windows 或驱动做了下混；
- VoiceMeeter 改变了声道顺序；
- 游戏把大量声音混到 C 声道；
- 游戏自身音频策略不是标准扬声器 panning。

### 5.3 默认忽略 C 声道会影响正前方判断

代码默认忽略 C 和 LFE。忽略 LFE 很合理，因为低频声道通常没有方向价值。但忽略 C 有利有弊。

好处是可以避免 UI、音乐、环境音被误判为正前方。坏处是如果游戏真的把正前方脚步或枪声主要输出到 C 声道，那么这些信息会被丢掉，导致正前方定位变差。

### 5.4 noise floor 可能被瞬态声音抬高

当前 noise floor 使用 attack / release 更新。强瞬态声音可能快速抬高 noise floor，导致紧随其后的连续脚步或枪声 SNR 降低，从而漏检。

更稳妥的做法是用 rolling median、percentile，或者在 onset 触发帧暂停更新 noise floor。

### 5.5 onset 使用 mono mixdown，损失方向信息

当前 onset 检测先把多声道混成 mono，再计算 spectral flux。这虽然简单，但会损失“哪个声道突增”的信息。

更好的做法是 per-channel spectral flux，或者 per-channel + per-band 的 onset 检测。这样事件检测阶段就能保留方向线索。

### 5.6 灵敏度滑杆语义容易混乱

代码注释中提到“滑杆值越高越敏感”，但实际公式和文档表现更像“值越高阈值越高，越不敏感”。这会让用户调参时困惑。

建议明确区分：

- Sensitivity：数值越高越容易触发；
- Noise rejection：数值越高越不容易误报。

当前实现更像后者。

### 5.7 线程参数更新不够规范

SNR threshold 使用 lock 更新，但 onset detector 的 `_flux_multiplier` 被 UI 线程直接修改，而处理线程也会读取它。这通常不一定崩溃，但违反了文档中“UI 与 processing 通过只读参数快照同步”的设计边界。

## 6. 改进建议

### 6.1 优先补日志和错误提示

这是最重要的工程改进。建议：

- 替换所有无提示的 `except Exception: pass`；
- 音频回调记录异常和状态；
- processing loop 记录异常；
- UI 状态栏显示捕获状态、设备名、声道数、丢帧数；
- 捕获失败时列出可用音频设备；
- 声道数不匹配时明确提示用户检查 Windows / VoiceMeeter / 游戏设置。

### 6.2 增加多声道链路自检

建议主程序启动时主动检查：

- 当前使用的输出设备；
- 设备支持的最大输出声道；
- 配置期望声道数；
- 实际捕获帧的声道数；
- 是否存在明显下混。

还可以提供一个声道测试模式，让用户播放 L / R / C / Ls / Rs / Lb / Rb 测试音并验证方向映射。

### 6.3 改进 onset 检测

建议从 mono onset 改成：

- per-channel onset；
- per-band onset；
- 每个频段独立 refractory；
- 每个方向或声道独立 refractory。

这样可以减少重叠声源和连续声音导致的漏检。

### 6.4 改进 noise floor

建议：

- 使用 rolling median 或 percentile 估计 noise floor；
- onset 触发帧不更新 noise floor；
- 将环境噪声 floor 和短期能量 envelope 分开；
- 对不同频段分别维护 noise floor。

这样可以避免强瞬态声音污染噪声底。

### 6.5 改进方向估计

建议把方向估计改成更严格的相邻声道对算法：

1. 按角度排序所有可用声道；
2. 构造相邻声道 pair；
3. 为每个 pair 计算总 SNR 或 pair score；
4. 选得分最高的 pair；
5. 在 pair 内根据能量比例插值；
6. 是否使用 C 声道由配置决定。

这会比“最高 SNR + 最近声道”更稳定，也更符合文档中的 VBAP 思路。

### 6.6 支持多声源候选

当前每帧只输出一个方向。可以改为：

- 找多个局部峰值；
- 每个峰值生成一个候选方向；
- 用 NMS 或角度间隔过滤；
- UI 同时显示多个 blip；
- 对每个方向分别做时间平滑。

这对真实游戏场景会更有意义。

### 6.7 完善测试

建议增加以下测试：

- `OnsetDetector` 的触发、阈值和 refractory 测试；
- 配置加载和 profile 合并测试；
- 缺失配置字段测试；
- 设备错误路径测试；
- 多声源和非相邻声道场景测试；
- 灵敏度滑杆参数更新测试；
- calibration 输出格式测试。

## 7. 与 HRTF 问题的关系

这个仓库的设计思路和“HRTF 为什么在某些游戏反而降低精度”高度相关。

HRTF 立体声虽然更接近双耳听觉，但它把方向信息编码成复杂的频谱变化、延迟和耳廓滤波。不同人耳廓差异、游戏混响、动态压缩、多声源叠加都会让反解变得困难。

`sound_radar` 选择绕开 HRTF，让游戏输出 7.1 / 5.1 离散声道。这样方向信息更直接地体现在不同声道能量上，算法不需要理解复杂的双耳频谱，只需要分析多声道能量分布。

但这也带来新的限制：它必须依赖真实可靠的多声道输出链路。如果游戏或系统没有正确输出离散多声道，这个方案也会失效。

## 8. 安全与合规提醒

这类声音雷达工具虽然从工程角度可以作为音频分析和可视化实验，但在多人竞技游戏中可能违反游戏服务条款，甚至触发反作弊风险。

因此更适合用于：

- 音频处理学习；
- 多声道分析实验；
- PyQt 可视化练习；
- Windows WASAPI loopback 研究；
- 游戏音频设计理解。

不建议在正式多人竞技环境中实际使用。

---

## 9. 最终评价

`sound_radar` 是一个思路清晰的音频分析原型。它避开 HRTF 反解，利用 7.1 / 5.1 多声道能量分布来估计声音方向，这个方向在理论上比直接分析耳机立体声更稳定。

但当前代码还没有达到文档中描述的完整程度。它已经实现了基本 pipeline，但工程可靠性、错误提示、配置兜底、onset 精度、多声源处理和方向估计算法都还有较大改进空间。

可以把它理解为：

> 一个已经能跑通的“多声道声音雷达 MVP”，而不是一个成熟稳定的实战级工具。
